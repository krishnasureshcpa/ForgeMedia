1.1  PRODUCT THESIS

ForgeMedia is positioned as a privacy-first, offline-native macOS media
command center — not "another video converter." The addressable user base
spans five distinct personas, all sharing a common need: serious media
processing (transcode, stitch, transcribe, restore, batch) without cloud
dependency, UI friction, or job fragility.

  TARGET USER PERSONAS:

  [1] Content Creators / YouTubers
      - Need fast exports, captions, batch processing, reliable progress,
        and minimal technical setup.
      - Pain: professional tools are powerful but slow/complex; consumer
        tools are simple but limited.

  [2] Podcasters / Interview Producers
      - Need transcription, subtitles, dubbing, audio cleanup, and
        repeatable presets.
      - Pain: transcription and subtitle alignment are often manual, slow,
        or cloud-dependent.

  [3] Archivists / Families / Small Studios
      - Need to process old footage, long videos, legacy formats, and
        preserve quality.
      - Pain: long jobs crash, quality settings are confusing, and output
        validation is weak.

  [4] Prosumers / Filmmakers
      - Need high-quality conversion, stitching, color/HDR awareness, and
      - control over codecs.
      - Pain: they want FFmpeg-level control without FFmpeg-level friction.

  [5] Privacy-Conscious Users
      - Need offline processing for sensitive media.
      - Pain: AI transcription/dubbing often requires uploads or
        subscriptions.

  PRODUCT POSITIONING STATEMENT:

  > A privacy-first, local, premium media command center for long-form
  > creators, archivists, and studios who want fast, trustworthy, offline
  > processing with an Apple-native interface.

  CORE CUSTOMER PROMISE:

  > Your media stays on your Mac. No telemetry. No analytics. No cloud
  > uploads by default.


1.2  MECHANICAL EMPATHY FRAMEWORK — APPLE SILICON MAPPING

The architecture is designed with explicit asymmetric-core awareness to
maximize performance-per-watt and prevent thermal throttling:

  EFFICIENCY CORES (Icestorm):
  - GRDB SQLite queue operations
  - Metadata probing
  - UI state observation via ValueObservation
  - Job event persistence
  - Agent routing decisions
  These are latency-tolerant, high-frequency-low-intensity workloads.

  PERFORMANCE CORES (Firestorm):
  - FFmpeg process execution
  - AVFoundation AVMutableComposition assembly
  - Whisper.cpp transcription (ARM NEON / Accelerate)
  - Ollama local inference dispatch
  These occupy short bursts with high throughput demand.

  APPLE NEURAL ENGINE (ANE):
  - Reserved exclusively for Core ML .mlpackage inference paths
    (Whisper encoder, upscaling, dubbing)
  - Treated as an optimisation fallback, never the primary path
  - ANE availability and model compatibility are inconsistent across
    M-series generations

  GPU (Metal):
  - Metal Performance Shaders for upscaling pipelines
  - Whisper Metal backend
  - Future Core ML compute paths
  - Resource budget tracks GPU busy state before dispatching any
    GPU-bound agent work

  THERMAL STRATEGY:
  The menu bar UI (MenuBarExtra) is architecturally forbidden from
  executing media work. Heavy jobs are dispatched to a background worker
  process or XPC helper, ensuring the UI thread never contributes to
  thermal pressure.

2.1  EMOTIONAL & NEUROLOGICAL ANCHORS

ForgeMedia's brand psychology targets calm capability — the opposite of
"neon chrome gaming converter" energy. The emotional vector is:

  > "My media is safe, the app is working, I can trust it with a 5-hour
  > job and walk away."

Key psychological decisions:

  NO DARK BASE
  Eliminates the "power user tool" cliché. Signals approachability,
  cleanliness, professionalism. Explicit instruction from the ShiftMedia
  legacy redesign notes.

  NO COLOR-AS-BACKGROUND
  Color is reserved for affordance and state:
    - Accent (--fm-accent #0066cc) for primary action per panel
    - Green (--fm-success #1f8f55) for completion
    - Amber (--fm-warning #b7791f) for warnings
    - Red (--fm-danger #c43b3b) for failures
  Never decoration. Never gradient-as-brand.

  NEUTRAL MATERIAL HIERARCHY
  Creates depth through lightness, not shadow or gradient:
    - Background: #f5f5f7
    - Surface: #ffffff
    - Surface raised: #fbfbfd
    - Border soft: #e8e8ed
    - Border: #d2d2d7

  RESTRAINED ACCENT
  Apple blue #0066cc used only for:
    - One primary action per panel
    - One progress signal per job
  This prevents visual noise and preserves scannability.

2.3  FLUID MOTION & INTERFACE FUNDAMENTALS

Motion is state-reactive, not decorative. Every animation serves a
specific communicative purpose.

  APPROVED MOTION PATTERNS:

  Pattern              | Duration  | Behaviour
  ---------------------|-----------|-----------------------------------
  Button press         | 80–120ms  | Tiny scale 0.98, no bounce
  Panel entry          | 220ms     | Ease-out, opacity + Y translation
  Progress fill        | 150–300ms | Smooth, never flashing
  Phase reveal         | 180–250ms | Only when phase actually changes
  Success check        | 180ms     | One-shot, no looping sparkle
  Error shake          | 180–220ms | Use sparingly; pair with error text
  Drag morph (spring)  | resp 0.35 | dampingFraction 0.86

  FREQUENCY RULES:
  - Frequent interactions: nearly instant (<200ms)
  - Occasional panels: 200–350ms
  - Rare completion moments: richer motion allowed

  REDUCED MOTION:
  Must remove all transform/scale/rotate animations.
  Preserve opacity/color transitions only when state-critical.
  Use @media (prefers-reduced-motion: reduce) / SwiftUI equivalent.

  EXPLICITLY AVOID:
  - Endless spinners for media jobs
  - Decorative looping gradients
  - Motion as the only indication of state
  - Progress bars that imply false precision
  - Skeleton shimmer after content is available
  - Framer-style black void as main surface
  - "Working..." / "Almost done..." placeholder copy

2.5  STATE COVERAGE — 11 JOB LIFECYCLE STATES

Every user-facing surface must handle:

  1. Idle / ready    — media loaded, waiting for action
  2. Preparing       — probing, estimating, building segment plan
  3. Running         — work underway with measured/estimated progress
  4. Taking longer   — phase exceeds 15s threshold with fallback copy
  5. Paused          — user or system pause with resume point
  6. Completed       — success with output location and checks
  7. Completed w/ warnings — usable but non-blocking issues found
  8. Failed          — unrecoverable with cause and next action
  9. Canceled        — user-initiated stop with cleanup report
  10. Recovered      — post-crash/relaunch with partial output options
  11. Edge           — long filenames, missing metadata, RTL text,
                       low disk space

Three feedback surfaces:
  - Menu bar: compact phase + progress + pause/cancel
  - Main window: richer job timeline + quality checks + recovery
  - Processing sheet: step list + engine progress + collapsed logs

3.4  UNORTHODOX RESOURCE UTILIZATION

  CHUNKED PROCESSING FOR LONG VIDEOS
  - Probe first → split into resumable segments → process individually
  - Preserve checkpoints in GRDB after each segment
  - Avoid loading 5-hour video entirely into memory
  - Partial output recovery on crash or cancellation

  THROTTLED UI PROGRESS
  - ValueObservation frequency deliberately limited
  - Prevents SwiftUI churn during rapid FFmpeg progress events
  - Progress confidence labels: unknown | estimated | measured

  CANCELLATION PROPAGATION
  - Process.interrupt() / Process.terminate() for child processes
  - GRDB checkpoint persistence enables resume
  - Temporary file cleanup on cancellation

  PRIVACY-FIRST NETWORKING
  - No analytics SDK initialisation
  - No telemetry endpoints compiled in
  - No automatic crash upload
  - Network client conditionally compiled
  - Gated behind explicit PrivacySettings flags

4.1  ZERO-COST STATE ENGINE

Job lifecycle is a discriminated union (JobPhase enum) with 11 states:

  idle -> preparing -> probing -> planning -> running -> validating ->
    completed | completedWithWarnings | failed | paused | canceled |
    recovered

KEY PROPERTIES:
  - progressConfidence (unknown | estimated | measured) tracked
    independently of progressFraction to prevent false precision
  - UI views are pure projections of domain state
  - No side effects in views
  - No hidden mutations


4.2  ZERO-I/O-BOTTLENECK PERSISTENCE

  Database: GRDB DatabaseQueue
  - Single-user local app configuration
  - ValueObservation for async Swift streaming (not Combine, not RxSwift)
  - Migrations for all schema changes

  INITIAL MIGRATION: "2026061301-initial-schema"
  - Creates presets, jobs, job_events tables
  - Includes rollback for each migration


4.3  ASYNCHRONOUS PROCESSING LAYER

  @MainActor          — all SwiftUI views
  actor isolation     — database writer, resource budget, cancellation
                        coordinator
  async functions     — all Process launches, progress parsing
  Task                — background worker dispatch with cooperative
                        cancellation
  AsyncSequence       — progress event streaming from child processes


4.4  PROGRESS FEEDBACK ARCHITECTURE

  Copy before percentage:

    "Transcribing segment 8 of 14…"  — good
    "Working…"                       — bad

  Progress confidence labels:

    "Estimated"   — based on file size, segment count, or local timing
    "Measured"    — based on completed segments or engine progress
    "Unknown"     — engine cannot report progress yet
    "Validating"  — work complete but output checks running

  Error recovery rules:

    - First retry: immediate
    - Second/third: exponential backoff (2s, 4s, 8s max)
    - After 3 failures: replace "Retry" with "Contact support"
      plus copyable error ID
    - Never clear user input on failure
    - Never blame the user for engine failures

│  DESIGN & ENGINEERING CONSTRAINTS (NON-NEGOTIABLE)                      │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘

  1. No dark base. Visual foundation is #f5f5f7 light neutral. Colour
     is state signal, not background.

  2. No telemetry/analytics by default. Privacy On is the default and
     the selling point.

  3. Menu bar must stay responsive. Architectural prohibition on media
     work in UI thread.

  4. 5-hour video must not crash. Chunked, resumable, checkpointed
     processing is mandatory.

  5. Output quality must be validated. Checksums, duration matching,
     codec/bitrate/audio verification per job.

  6. AVFoundation first, FFmpeg second. Use native Apple paths where
     possible. FFmpeg for edge cases only.

  7. ANE is a fallback, not a promise. Core ML acceleration is an
     optimisation, never a guarantee.

  8. Privacy prompts must be explicit. "Privacy On: local processing"
     language, never "we may use your data."

  9. One accent per surface. Sparse colour usage prevents visual noise.

  10. Reduced motion must be respected. No transform/scale/rotate
      animation when prefers-reduced-motion is active.
